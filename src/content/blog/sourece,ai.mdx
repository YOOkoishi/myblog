### 第一部分：提问的艺术（30分钟）

#### 为什么要学会提问？

在技术学习过程中，提问是获取知识最快的方式之一。但是，**好的提问**和**糟糕的提问**效果天差地别。

#### ❌ 糟糕的提问示例

```text
"我的代码报错了，怎么办？"
"为什么运行不了？"
"Linux太难了，我不会用"
```

**问题所在：**
- 没有提供错误信息
- 没有说明具体环境
- 没有描述已经尝试过的解决方法

#### ✅ 优秀的提问示例

```text
【问题】在 Ubuntu 22.04 上编译 C++ 程序时出现链接错误

【环境信息】
- 操作系统：Ubuntu 22.04 LTS
- 编译器：g++ 11.4.0
- 项目：使用 CMake 3.22.1

【错误信息】
undefined reference to `std::cout'

【已尝试的解决方法】
1. 检查了 #include <iostream>
2. 确认了使用 std:: 前缀
3. 清理并重新编译

【问题】
怀疑是链接器配置问题，CMakeLists.txt 应该如何配置？
```

#### 提问的黄金法则

1. **先自己搜索**
   - 使用搜索引擎（Google、Bing、百度）
   - 查看官方文档
   - 搜索 Stack Overflow、GitHub Issues

2. **描述清楚你的环境**
   - 操作系统版本
   - 软件/工具版本
   - 相关配置信息

3. **提供完整的错误信息**
   - 完整的错误日志
   - 截图（如果必要）
   - 最小可复现代码

4. **说明你已经尝试过什么**
   - 展示你的思考过程
   - 列出尝试过的解决方案

5. **礼貌和尊重**
   - 使用"请问"、"谢谢"等礼貌用语
   - 对帮助你的人表示感谢

#### 推荐阅读

- [《提问的智慧》](https://github.com/ryanhanwu/How-To-Ask-Questions-The-Smart-Way/blob/main/README-zh_CN.md)
- [《别像弱智一样提问》](https://github.com/tangx/Stop-Ask-Questions-The-Stupid-Ways)

### 第二部分：Linux Shell 基础（45分钟）

#### 什么是 Shell？

Shell 是用户与 Linux 系统交互的命令行界面。常见的 Shell 有：
- **Bash**（最常用）
- Zsh（功能更强大）
- Fish（用户友好）

#### 基础命令速查

##### 文件和目录操作

```bash
# 查看当前目录
pwd

# 列出文件
ls          # 简单列出
ls -l       # 详细信息
ls -la      # 包含隐藏文件
ls -lh      # 人类可读的文件大小

# 切换目录
cd /path/to/directory
cd ~        # 回到家目录
cd ..       # 上一级目录
cd -        # 上一次所在目录

# 创建目录
mkdir mydir
mkdir -p parent/child/grandchild  # 递归创建

# 删除文件和目录
rm file.txt           # 删除文件
rm -r directory       # 递归删除目录
rm -rf directory      # 强制删除（危险！）

# 复制和移动
cp source.txt dest.txt       # 复制文件
cp -r source_dir dest_dir    # 复制目录
mv oldname.txt newname.txt   # 重命名/移动
```

##### 文件查看和编辑

```bash
# 查看文件内容
cat file.txt        # 直接输出全部内容
less file.txt       # 分页查看（推荐）
head file.txt       # 查看前10行
tail file.txt       # 查看后10行
tail -f log.txt     # 实时查看日志

# 文本编辑器
nano file.txt       # 简单易用
vim file.txt        # 强大但需要学习
```

##### 文件搜索

```bash
# 查找文件
find . -name "*.txt"              # 按文件名查找
find . -type f -size +10M         # 查找大于10M的文件

# 搜索文件内容
grep "keyword" file.txt           # 在文件中搜索
grep -r "keyword" directory       # 递归搜索目录
grep -i "keyword" file.txt        # 忽略大小写
```

##### 权限管理

```bash
# 查看权限
ls -l

# 修改权限
chmod 755 script.sh      # rwxr-xr-x
chmod +x script.sh       # 添加执行权限
chmod -w file.txt        # 移除写权限

# 修改所有者
sudo chown user:group file.txt
```

##### 进程管理

```bash
# 查看进程
ps aux              # 查看所有进程
top                 # 实时监控（按 q 退出）
htop                # 更好用的 top（需要安装）

# 终止进程
kill PID            # 正常终止
kill -9 PID         # 强制终止
killall process_name
```

##### 网络操作

```bash
# 下载文件
wget https://example.com/file.zip
curl -O https://example.com/file.zip

# 网络连接
ping google.com
ifconfig            # 查看网络接口（旧）
ip addr             # 查看网络接口（新）
```

#### Shell 脚本入门

创建一个简单的 Shell 脚本：

```bash
#!/bin/bash
# 这是一个注释

echo "Hello, World!"
echo "当前用户: $USER"
echo "当前目录: $PWD"

# 变量
NAME="张三"
echo "你好, $NAME"

# 条件判断
if [ -f "file.txt" ]; then
    echo "文件存在"
else
    echo "文件不存在"
fi

# 循环
for i in {1..5}; do
    echo "循环第 $i 次"
done
```

保存为 `script.sh`，然后：

```bash
chmod +x script.sh
./script.sh
```

#### 实用技巧

##### 管道和重定向

```bash
# 管道 | : 将前一个命令的输出作为后一个命令的输入
ls -l | grep ".txt"
cat file.txt | wc -l          # 统计行数

# 重定向
command > output.txt          # 覆盖写入
command >> output.txt         # 追加写入
command 2> error.log          # 错误输出重定向
command > output.txt 2>&1     # 标准输出和错误都重定向
```

##### 快捷键

- `Ctrl + C`：终止当前命令
- `Ctrl + Z`：暂停当前命令
- `Ctrl + D`：退出当前 Shell
- `Ctrl + L`：清屏（等同于 `clear`）
- `Ctrl + R`：搜索命令历史
- `Tab`：自动补全
- `!!`：执行上一条命令
- `!$`：上一条命令的最后一个参数

### 第三部分：Git 与 GitHub（45分钟）

#### 什么是 Git？

Git 是一个**分布式版本控制系统**，用于跟踪代码变更、协作开发。

**核心概念：**
- **仓库（Repository）**：存储项目的地方
- **提交（Commit）**：保存代码快照
- **分支（Branch）**：独立的开发线
- **远程（Remote）**：托管在服务器上的仓库

#### Git 基础命令

##### 初始配置

```bash
# 设置用户信息
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 查看配置
git config --list
```

##### 创建仓库

```bash
# 初始化新仓库
git init

# 克隆现有仓库
git clone https://github.com/user/repo.git
```

##### 基本工作流

```bash
# 1. 查看状态
git status

# 2. 添加文件到暂存区
git add file.txt        # 添加单个文件
git add .               # 添加所有修改

# 3. 提交更改
git commit -m "Add feature: user login"

# 4. 查看历史
git log
git log --oneline       # 简洁模式
git log --graph         # 图形化显示分支
```

##### 分支操作

```bash
# 查看分支
git branch

# 创建分支
git branch feature-login

# 切换分支
git checkout feature-login
# 或者（Git 2.23+）
git switch feature-login

# 创建并切换
git checkout -b feature-login
# 或者
git switch -c feature-login

# 合并分支
git checkout main
git merge feature-login

# 删除分支
git branch -d feature-login
```

##### 远程操作

```bash
# 查看远程仓库
git remote -v

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 推送到远程
git push origin main
git push -u origin main     # 第一次推送，设置上游

# 从远程拉取
git pull origin main

# 获取远程更新（不合并）
git fetch origin
```

##### 撤销操作

```bash
# 撤销工作区修改
git checkout -- file.txt
# 或者
git restore file.txt

# 撤销暂存区文件
git reset HEAD file.txt
# 或者
git restore --staged file.txt

# 撤销最后一次提交（保留更改）
git reset --soft HEAD~1

# 撤销最后一次提交（丢弃更改）
git reset --hard HEAD~1
```

#### GitHub 使用指南

##### 创建仓库

1. 登录 GitHub
2. 点击右上角 `+` → `New repository`
3. 填写仓库名称和描述
4. 选择公开或私有
5. 可选：添加 README、.gitignore、LICENSE

##### Fork 和 Pull Request 流程

这是参与开源项目的标准流程：

```bash
# 1. Fork 项目到自己的账号（在 GitHub 网页上点击 Fork）

# 2. 克隆你 Fork 的仓库
git clone https://github.com/your-username/repo.git
cd repo

# 3. 添加原仓库为上游
git remote add upstream https://github.com/original-owner/repo.git

# 4. 创建新分支进行修改
git checkout -b fix-typo

# 5. 进行修改，然后提交
git add .
git commit -m "Fix typo in README"

# 6. 推送到你的 Fork
git push origin fix-typo

# 7. 在 GitHub 上创建 Pull Request
#    访问你的 Fork 页面，点击 "Compare & pull request"
```

##### 保持 Fork 同步

```bash
# 拉取上游更新
git fetch upstream

# 切换到主分支
git checkout main

# 合并上游更新
git merge upstream/main

# 推送到你的 Fork
git push origin main
```

#### Git 最佳实践

1. **频繁提交，但要有意义**
   ```bash
   ✅ git commit -m "Add user authentication module"
   ❌ git commit -m "update"
   ```

2. **写清晰的提交信息**
   ```
   格式：<type>: <subject>
   
   feat: Add user login feature
   fix: Resolve memory leak in parser
   docs: Update installation guide
   style: Format code with prettier
   refactor: Simplify database query logic
   test: Add unit tests for API
   chore: Update dependencies
   ```

3. **使用 .gitignore**
   ```gitignore
   # Python
   __pycache__/
   *.pyc
   venv/
   
   # Node.js
   node_modules/
   npm-debug.log
   
   # IDE
   .vscode/
   .idea/
   
   # OS
   .DS_Store
   Thumbs.db
   ```

4. **永远不要提交敏感信息**
   - 密码、API密钥
   - 个人身份信息
   - 数据库连接字符串

### 第四部分：如何给开源社区做贡献（30分钟）

#### 为什么参与开源？

- 📚 **学习**：阅读优秀代码，提升技能
- 🤝 **社交**：结识志同道合的开发者
- 💼 **简历**：展示你的实际项目经验
- 🌟 **回馈**：帮助你使用过的项目变得更好

#### 从哪里开始？

##### 1. 选择合适的项目

**新手友好的项目特征：**
- 有 `good first issue` 或 `beginner-friendly` 标签
- 文档完善
- 社区活跃
- 维护者友好

**推荐项目类型：**
- 你正在使用的工具
- 你感兴趣的领域
- 文档或翻译项目（不需要太多编程）

##### 2. 非代码贡献

**你不一定要写代码才能贡献！**

- 📝 **改进文档**：修正错别字、补充说明
- 🌏 **翻译**：将文档翻译成中文
- 🐛 **报告 Bug**：提供详细的错误报告
- 💡 **建议功能**：提出有价值的想法
- ✅ **测试**：帮助测试新功能
- ❓ **回答问题**：在 Issues 中帮助其他用户

##### 3. 代码贡献流程

**完整的贡献流程：**

```
1. 选择 Issue
   ↓
2. Fork 项目
   ↓
3. 克隆到本地
   ↓
4. 创建分支
   ↓
5. 进行修改
   ↓
6. 运行测试
   ↓
7. 提交更改
   ↓
8. 推送到 Fork
   ↓
9. 创建 Pull Request
   ↓
10. 代码审查
   ↓
11. 修改和完善
   ↓
12. 合并！🎉
```

##### 4. 你的第一个 PR

**示例：为一个项目修复文档错误**

```bash
# 1. Fork 项目并克隆
git clone https://github.com/your-username/awesome-project.git
cd awesome-project

# 2. 创建分支
git checkout -b docs/fix-readme-typo

# 3. 修改文件（例如 README.md）
# 将 "instalation" 改为 "installation"

# 4. 提交
git add README.md
git commit -m "docs: Fix typo in installation section

Changed 'instalation' to 'installation' in README.md"

# 5. 推送
git push origin docs/fix-readme-typo

# 6. 在 GitHub 上创建 PR
```

**PR 描述模板：**

```markdown
## 描述
修复了 README.md 中的拼写错误

## 修改内容
- 将 "instalation" 改为 "installation"

## 相关 Issue
Fixes #123

## 检查清单
- [x] 我已经阅读了贡献指南
- [x] 我的修改遵循项目的代码规范
- [x] 我已经测试了我的修改
```

#### 贡献时的注意事项

##### ✅ 应该做的

1. **先阅读贡献指南**
   - 查看 `CONTRIBUTING.md`
   - 遵循项目的代码规范

2. **与维护者沟通**
   - 在开始前在 Issue 中说明你要做什么
   - 避免重复劳动

3. **保持 PR 简洁**
   - 一个 PR 只做一件事
   - 不要混合多个不相关的修改

4. **及时响应反馈**
   - 查看 Code Review 的评论
   - 及时修改和完善

##### ❌ 不应该做的

1. **不要随意修改不相关的文件**
2. **不要提交垃圾代码**
3. **不要在 PR 中加入无关的格式化修改**
4. **不要忽视 CI 测试失败**
5. **不要催促维护者**（他们通常是志愿者）
